Voici les directives techniques strictes à appliquer sur vos infrastructures pour combler les failles découvertes :

    {% if detect.ffuf %}
  • 🚨 [CRITIQUE — URGENCE] Fuite d'identifiants / Fichiers sensibles exposés (FFUF)
    Des identifiants ou fichiers de configuration critiques sont accessibles publiquement sur le serveur web.
    • Dans l'heure : Changer le mot de passe MySQL exposé — ALTER USER 'user'@'localhost' IDENTIFIED BY '[NouveauMDP]'; FLUSH PRIVILEGES;
    • Supprimer le fichier : rm -f /var/www/html/config.bak && find /var/www/html -name '*.bak' -o -name '*.env' | xargs rm -f
    • Déplacer hors du webroot : mv /var/www/html/config* /etc/monapp/ && chmod 600 /etc/monapp/*
    • Bloquer via .htaccess : <FilesMatch "\.(bak|env|sql|old|conf)$"> Deny from all </FilesMatch>
    • Audit d'exploitation : grep 'config.bak' /var/log/apache2/access.log | wc -l — Si accès détectés, isoler le serveur et révoquer toutes les sessions actives.
    • Prévention CI/CD : Ajouter *.bak, *.env au .gitignore et intégrer FFUF dans le pipeline.
    • {% endif %} {% if detect.nmap %}
  • 🔓 [RÉSEAU] Ports sensibles exposés sur Internet (Nmap)
    Des ports d'administration (SSH, MySQL, RDP) sont accessibles directement depuis l'Internet public.
    • Pare-feu (par défaut DENY) : ufw default deny incoming && ufw allow from [IP_ADMIN] to any port [PORT]
    • SSH (port 22) : PasswordAuthentication no + PermitRootLogin no dans /etc/ssh/sshd_config. Clés Ed25519 uniquement : ssh-keygen -t ed25519
    • Anti-bruteforce SSH : apt install fail2ban — configurer maxretry=3, bantime=86400 dans /etc/fail2ban/jail.local
    • MySQL (port 3306) : Lier à localhost uniquement — bind-address = 127.0.0.1 dans /etc/mysql/mysql.conf.d/mysqld.cnf. Accès distant via tunnel SSH uniquement.
    • Vérifier les logs : tail -f /var/log/auth.log | grep 'Failed password'
    • {% endif %} {% if detect.config or detect.outdated %}
  • ⚙️ [SERVEUR] Mauvaises configurations et composants obsolètes (Nikto)
    Le serveur divulgue sa version technique ou présente des défauts de durcissement.
    • Masquer la version Apache : ServerTokens Prod + ServerSignature Off dans /etc/apache2/conf-available/security.conf
    • Masquer la version Nginx : server_tokens off; dans le bloc http {} de nginx.conf
    • Mettre à jour : apt update && apt upgrade apache2 nginx php -y
    • Désactiver les modules inutiles : a2dismod autoindex status info
    • Interdire la navigation dans les répertoires : Options -Indexes dans .htaccess
    • robots.txt : Ne jamais y lister des répertoires confidentiels. Protéger chaque répertoire mentionné avec un contrôle d'accès IP.
    • {% endif %} {% if detect.ssl %}
  • 🔒 [CRYPTOGRAPHIE] Certificat SSL/TLS invalide ou protocoles obsolètes (TestSSL)
    La chaîne de confiance TLS est brisée ou des protocoles/ciphers faibles sont activés.
    • Let's Encrypt (gratuit) : apt install certbot python3-certbot-apache && certbot --apache -d [domaine]
    • Renouvellement auto : echo "0 3 * * * certbot renew --quiet" | crontab -
    • Désactiver TLS 1.0/1.1 : SSLProtocol -all +TLSv1.2 +TLSv1.3 (Apache) / ssl_protocols TLSv1.2 TLSv1.3; (Nginx)
    • Supprimer les ciphers faibles : SSLCipherSuite HIGH:!aNULL:!MD5:!3DES:!RC4:!EXP
    • Activer HSTS : Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
    • Valider : ssllabs.com/ssltest/ — Score cible : A+
    • {% endif %} {% if detect.headers %}
  • 🛡️ [APPLICATIF] En-têtes de sécurité HTTP manquants (Nikto / ZAP)
    L'application est vulnérable aux attaques XSS, Clickjacking et autres attaques côté client.
    • Activer le module : a2enmod headers && systemctl restart apache2
    • X-Content-Type-Options : Header always set X-Content-Type-Options "nosniff"
    • X-Frame-Options : Header always set X-Frame-Options "SAMEORIGIN"
    • XSS Protection : Header always set X-XSS-Protection "1; mode=block"
    • Referrer-Policy : Header always set Referrer-Policy "strict-origin-when-cross-origin"
    • Content-Security-Policy : Header always set Content-Security-Policy "default-src 'self'; script-src 'self'"
    • Vérifier : curl -I https://[domaine] — Score cible sur securityheaders.com : A+
    • {% endif %} {% set has_zap = report.scoring_anssi.get('Critique', [])|selectattr('outil', 'equalto', 'OWASP ZAP')|list|length > 0 or report.scoring_anssi.get('Majeur', [])|selectattr('outil', 'equalto', 'OWASP ZAP')|list|length > 0 %} {% if has_zap %}
  • 🕸️ [APPLICATIF WEB] Vulnérabilités détectées par OWASP ZAP
    Des failles applicatives actives ont été confirmées par l'attaque automatisée ZAP.
    • [SQL Injection] : Utiliser des requêtes préparées — $pdo->prepare("SELECT * FROM users WHERE id = ?") — jamais de concaténation.
    • [XSS] : Encoder toutes les sorties — htmlspecialchars($var, ENT_QUOTES, 'UTF-8'). Implémenter un CSP strict.
    • [CSRF] : Ajouter un token CSRF sur chaque formulaire POST. Valider côté serveur à chaque requête.
    • [Auth faible] : Sessions avec HttpOnly + Secure + SameSite=Strict. Implémenter le MFA.
    • [Exposition info] : Désactiver display_errors en production. Logger les erreurs dans un fichier privé.
    • Régression : Réintégrer ZAP dans le pipeline CI/CD et relancer après chaque correctif.
    • {% endif %}
